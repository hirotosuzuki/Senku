# フェーズ1: SQLパーサー - データベースの「通訳者」を作る

## はじめに

こんにちは、古の老賢者です。今日は、データベースエンジン開発の第一歩である「SQLパーサー」について語りましょう。

データベースを作るという壮大な旅路において、SQLパーサーは最初の重要な関門です。なぜなら、ユーザーが書いたSQL文という「人間の言葉」を、データベースエンジンが理解できる「機械の言葉」に翻訳する役割を担うからです。

まるで、異なる言語を話す人々の間で通訳をするようなものですね。この通訳者がいなければ、どんなに優れたデータベースエンジンも、ユーザーの意図を理解することができません。

## 何をやるのか：フェーズ1のSQLパーサーの役割

フェーズ1では、最小限の機能を持つSQLパーサーを実装します。具体的には、以下の3つのSQL文を理解できるようにします：

### 1. CREATE TABLE - テーブルの作成

```sql
CREATE TABLE users(id INT, name TEXT)
```

この文を解析して、「usersという名前のテーブルを作成し、id（整数型）とname（文字列型）という2つのカラムを持つ」という構造を理解します。

### 2. INSERT - データの挿入

```sql
INSERT INTO users VALUES (1, 'alice')
```

この文を解析して、「usersテーブルに、id=1、name='alice'というデータを挿入する」という操作を理解します。

### 3. SELECT - データの検索

```sql
SELECT * FROM users WHERE id = 1
```

この文を解析して、「usersテーブルから、idが1である行を全て取得する」というクエリを理解します。

## なぜやるのか：SQLパーサーの存在意義

### 1. 人間と機械の橋渡し

SQLパーサーは、人間が理解しやすい「宣言的な言語」であるSQLを、コンピューターが実行できる「命令的な操作」に変換します。

例えば、`SELECT * FROM users WHERE id = 1`というSQL文は、人間にとっては「idが1のユーザーを探して」という直感的な意味ですが、データベースエンジンにとっては以下のような一連の操作に変換する必要があります：

1. usersテーブルのファイルを開く
2. 各行を順番に読み込む
3. 各行のidカラムの値を確認する
4. idが1の行だけを結果セットに追加する
5. 結果セットを返す

この変換こそが、パーサーの仕事です。

### 2. データベースエンジンの「入口」

データベースエンジンは、以下のような階層構造になっています：

```
ユーザー（SQL文を書く）
    ↓
SQLパーサー（SQLを解析してASTに変換）
    ↓
実行エンジン（ASTを解釈して実際の操作を実行）
    ↓
ストレージ層（ディスクへの読み書き）
```

パーサーは、この階層の最上位に位置し、すべてのクエリが通る「入口」です。パーサーが正しく動作しなければ、その後の処理は一切実行されません。

### 3. 将来の拡張性

フェーズ1では最小限の機能しか実装しませんが、この基盤があれば、将来的に以下のような機能を追加することが容易になります：

- UPDATE文（データの更新）
- DELETE文（データの削除）
- JOIN（テーブルの結合）
- サブクエリ
- 集約関数（COUNT, SUM, AVGなど）
- トランザクション制御（BEGIN, COMMIT, ROLLBACK）

## 歴史的な背景：SQLとパーサーの進化

### SQLの誕生（1970年代）

SQL（Structured Query Language）は、1970年代にIBMの研究員であるドナルド・チャンバリンとレイモンド・ボイスによって開発されました。当時は「SEQUEL」という名前でしたが、商標の問題で「SQL」に変更されました。

SQLの革新的な点は、**宣言的な言語**であることです。つまり、「何をしたいか」を書くだけで、「どうやってやるか」はデータベースエンジンが考えてくれます。これは、それまでの手続き型の言語とは大きく異なるアプローチでした。

### コンパイラ理論の確立（1950-1960年代）

SQLパーサーの技術的基盤は、1950-1960年代に確立されたコンパイラ理論にあります。

コンパイラは、高級言語（人間が理解しやすい言語）を機械語（コンピューターが理解できる言語）に変換するプログラムです。この変換プロセスは、以下の3つの段階に分かれます：

1. **字句解析（Lexical Analysis）**
   - 文字列を意味のある最小単位（トークン）に分割
   - 例：「CREATE TABLE」→ `['CREATE', 'TABLE']`

2. **構文解析（Parsing）**
   - トークン列を構文規則に従って解析
   - 抽象構文木（AST: Abstract Syntax Tree）を生成

3. **意味解析・コード生成**
   - ASTを解釈して、実際の操作に変換

SQLパーサーも、この同じプロセスを踏みます。ただし、SQLの場合は「コード生成」の代わりに「実行計画の生成」を行います。

### パーサージェネレータの登場（1970-1980年代）

1970年代後半から1980年代にかけて、パーサージェネレータ（パーサーを自動生成するツール）が登場しました。代表的なものに：

- **yacc**（Yet Another Compiler Compiler, 1975年）
- **ANTLR**（ANother Tool for Language Recognition, 1989年）

これらのツールは、構文規則を定義するだけで、パーサーを自動生成してくれます。現代の多くのデータベースエンジン（PostgreSQL、MySQLなど）でも、これらのツールが使われています。

しかし、フェーズ1では、**手動でパーサーを実装**します。なぜなら、パーサージェネレータを使う前に、パーサーがどのように動作するかを理解することが重要だからです。基礎を理解せずにツールを使うのは、車の運転を覚える前に自動運転車に乗るようなものです。

## 技術的な詳細：パーサーの実装

### 1. 字句解析（Lexer）

字句解析は、SQL文をトークンに分割する処理です。

```python
class Lexer:
    def tokenize(self, line: str) -> List[str]:
        # "CREATE TABLE users(id INT, name TEXT)"
        # → ['CREATE', 'TABLE', 'users', '(', 'id', 'INT', ',', 'name', 'TEXT', ')']
```

字句解析器は、以下のような処理を行います：

- 空白や改行を無視
- キーワード（CREATE, TABLE, SELECTなど）を識別
- 識別子（テーブル名、カラム名）を識別
- リテラル（文字列、数値）を識別
- 演算子（=, >, <など）を識別
- クォートされた文字列を正しく処理

### 2. 構文解析（Parser）

構文解析は、トークン列を構文規則に従って解析し、ASTを生成する処理です。

```python
class SqlParser:
    def parse(self, line: str) -> ParsedStatement:
        toks = self.lexer.tokenize(line)
        # トークン列を解析してASTに変換
```

構文解析器は、以下のような処理を行います：

- トークン列がSQLの構文規則に従っているか確認
- 構文エラーがあれば、適切なエラーメッセージを返す
- 正しい構文であれば、ASTを生成

### 3. AST（抽象構文木）

ASTは、パースされたSQL文を表現するデータ構造です。

```python
@dataclass
class ParsedStatement:
    kind: StatementType  # CREATE, INSERT, SELECTなど
    table_name: str
    columns: List[ColumnDefinition]
    # ... その他の属性
```

ASTの利点は、**構造化されている**ことです。実行エンジンは、このASTを見るだけで、何をすべきかを理解できます。

## 実装のポイント

### 1. エラーハンドリング

パーサーは、不正なSQL文に対して適切なエラーメッセージを返す必要があります。例えば：

- `CREATE TABLE`（テーブル名がない）→ "CREATE TABLE statement requires table name"
- `SELECT FROM users`（カラム指定がない）→ "SELECT statement requires column list"

### 2. 拡張性

フェーズ1では最小限の機能しか実装しませんが、将来的な拡張を考慮した設計にします。例えば：

- 新しいSQL文（UPDATE, DELETE）を追加しやすい構造
- WHERE句の条件を拡張しやすい構造
- データ型を追加しやすい構造

### 3. テスト容易性

パーサーは、入力（SQL文）と出力（AST）が明確なので、テストが書きやすいです。各SQL文に対して、期待されるASTを定義し、実際の出力と比較することで、正しく動作しているかを確認できます。

## この技術で作れるプロダクト

SQLパーサーを理解すると、以下のようなプロダクトを作ることができます：

### 1. データベースエンジン

もちろん、データベースエンジンそのものです。PostgreSQL、MySQL、SQLiteなどのオープンソースデータベースは、すべてSQLパーサーを持っています。

### 2. クエリビルダー

ORM（Object-Relational Mapping）やクエリビルダーは、プログラムコードからSQLを生成しますが、その逆（SQLを解析してプログラムコードに変換）も可能です。

### 3. SQLフォーマッター

SQL文を整形するツール（Prettier for SQL）も、パーサー技術を使います。

### 4. SQLリントツール

SQL文の構文エラーや潜在的な問題を検出するツールも、パーサー技術を使います。

### 5. データベース管理ツール

phpMyAdmin、pgAdminなどのデータベース管理ツールも、SQLパーサーを使って、ユーザーが入力したSQL文を解析し、実行計画を表示したり、エラーを検出したりします。

## まとめ

SQLパーサーは、データベースエンジンの「入口」であり、人間と機械の橋渡しをする重要なコンポーネントです。

フェーズ1では、最小限の機能（CREATE TABLE, INSERT, SELECT）を実装しますが、この基盤があれば、将来的に様々な機能を追加することができます。

コンパイラ理論という古くからある技術を、現代のデータベースエンジン開発に応用する。これこそが、エンジニアとしての基礎力を身につける最良の方法です。

次回は、パーサーが生成したASTを実際に実行する「実行エンジン」について語りましょう。それでは、良いコーディングライフを！
