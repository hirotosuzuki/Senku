# フェーズ1: テーブル管理 - データベースの「目録」を作る

## はじめに

こんにちは、古の老賢者です。前回はストレージ層について語りましたが、今回は「テーブル管理」について語りましょう。

ストレージ層が「倉庫」なら、テーブル管理は「目録」です。倉庫に何が入っているか、どこに何があるかを記録する帳簿のようなものです。どんなに立派な倉庫があっても、目録がなければ、何がどこにあるかわからなくなってしまいます。

まるで、図書館で本を探すときに、目録（カタログ）がないと、何万冊もある本の中から目的の本を見つけられないのと同じですね。データベースも同じです。テーブルがいくつあるか、それぞれのテーブルにどんなカラムがあるか、どこに保存されているか。これらを管理するのが「テーブル管理」の役割です。

## 何をやるのか：フェーズ1のテーブル管理の役割

フェーズ1では、最小限の機能を持つテーブル管理を実装します。具体的には、以下の3つのコンポーネントを実装します：

### 1. Catalog - データベースカタログ

データベース内の全テーブルのメタデータを管理します。テーブル名、カラム情報、ファイルの場所などを記録します。

```python
catalog = Catalog(catalog_file)
catalog.create_table("users", schema, file_path)
```

カタログは、データベースの「目録」です。どのテーブルが存在するか、それぞれのテーブルにどんなカラムがあるかを記録します。

#### 余談：なぜ「Catalog」という名前なのか？

「カタログ」という言葉は、図書館や博物館の「目録」から来ています。データベースの世界では、1970年代にエドガー・コッドがリレーショナルデータベースの概念を提唱した際、メタデータを管理する仕組みを「システムカタログ」と呼びました。

PostgreSQLでは、`pg_class`、`pg_attribute`、`pg_type`などのシステムカタログテーブルが存在します。これらは、データベース内の全テーブル、カラム、型の情報を保持しています。

### 2. Schema - テーブルスキーマ

テーブルの構造（カラム名、データ型など）を定義します。SQLの`CREATE TABLE`文から生成され、カタログに登録されます。

```python
schema = Schema(
    table_name="users",
    columns=[
        ColumnDefinition(name="id", data_type="INT"),
        ColumnDefinition(name="name", data_type="TEXT"),
    ]
)
```

スキーマは、テーブルの「設計図」です。どのカラムがどんな型を持っているかを定義します。

### 3. Database - テーブル管理の統合

テーブルの作成、データの挿入・検索を統合的に管理します。カタログとストレージ層を橋渡しする役割を担います。

```python
db = Database()
db.execute("CREATE TABLE users (id INT, name TEXT)")
```

## なぜやるのか：テーブル管理の存在意義

### 1. メタデータの永続化

データベースを再起動しても、どのテーブルが存在するかを覚えていなければなりません。カタログは、この情報を永続化する役割を担います。

### 2. 名前解決

SQL文で`SELECT * FROM users`と書いたとき、`users`という名前から実際のテーブルファイルを見つける必要があります。カタログは、この「名前解決」を可能にします。

### 3. スキーマ検証

`INSERT INTO users VALUES (1, "alice")`と書いたとき、値がスキーマに適合するかを検証する必要があります。スキーマ情報があれば、この検証が可能になります。

### 4. 将来の拡張性

フェーズ1では最小限の機能しか実装しませんが、この基盤があれば、将来的に以下のような機能を追加することが容易になります：

- **主キー制約**: スキーマに主キー情報を追加
- **外部キー制約**: テーブル間の関係を管理
- **インデックス管理**: どのカラムにインデックスがあるかを記録
- **統計情報**: テーブルの行数、カラムの分布などを記録（クエリ最適化に使用）

## 歴史的な背景：メタデータ管理の進化

### 1960年代：メタデータの分散とその問題

1960年代の初期のデータベースシステムでは、メタデータ（テーブル名、カラム情報など）が分散していました。各アプリケーションが独自にメタデータを管理する必要があり、以下のような問題が発生していました：

- **情報の重複**: 同じテーブル情報が複数の場所に保存され、整合性が保てない
- **再起動時の情報喪失**: プログラムを終了すると、メタデータも失われてしまう
- **名前解決の困難**: `SELECT * FROM users`と書いても、`users`がどこにあるかわからない
- **スキーマ変更の複雑さ**: テーブル構造を変更する際、複数の場所を更新する必要がある

### 1970年代：メタデータの永続化と統一管理

1970年代になると、メタデータを永続化し、統一管理する必要性が認識されるようになりました。初期の解決策として、以下のようなアプローチが試みられました：

- **設定ファイル**: テーブル情報を設定ファイル（INI、XMLなど）に保存
- **専用のメタデータファイル**: データベース専用のメタデータファイルを作成
- **コード内での定義**: アプリケーションコード内でテーブル構造を定義

しかし、これらのアプローチには限界がありました。設定ファイルは構造化されていない、専用ファイルは問い合わせが困難、コード内定義は変更が大変などです。

### 1980年代：システムカタログの標準化

1980年代になると、「メタデータもリレーション（テーブル）として管理すべき」という重要な原則が確立されました。これにより、以下のようなメリットが得られました：

- **SQLで問い合わせ可能**: `SELECT * FROM pg_tables`のように、SQLでメタデータを取得できる
- **統一的な管理**: メタデータも通常のテーブルと同じように管理できる
- **動的なスキーマ変更**: `ALTER TABLE`でスキーマを変更し、カタログを自動更新

多くのデータベースシステムが「システムカタログ」を採用するようになりました：

- **IBM DB2**: `SYSCAT`スキーマにシステムカタログテーブルを配置
- **Oracle**: `SYS`スキーマに`DBA_TABLES`、`DBA_COLUMNS`などのビューを提供
- **PostgreSQL**: `pg_`で始まるシステムカタログテーブル（`pg_class`、`pg_attribute`など）

### 現代：システムカタログの拡張

現代のデータベースシステムでは、システムカタログは単なるメタデータ管理を超えて、以下のような機能を提供しています：

- **権限管理**: どのユーザーがどのテーブルにアクセスできるかを記録
- **統計情報**: テーブルの行数、カラムの分布などを記録（クエリ最適化に使用）
- **インデックス情報**: どのカラムにインデックスがあるかを記録
- **制約情報**: 主キー、外部キー、チェック制約などを記録

これらの実績により、システムカタログは「実証済みのアーキテクチャ」として確立されています。フェーズ1ではJSONファイルで実装していますが、将来的にはシステムカタログテーブルとして実装することで、SQLでメタデータを問い合わせできるようになります。

## 技術的な詳細：テーブル管理の実装

### 1. カタログの構造

カタログは、現在はJSONファイルで永続化しています。将来的には、システムカタログテーブルとして実装する予定です。

#### カタログファイルの構造

```json
{
  "tables": {
    "users": {
      "columns": [
        {"name": "id", "data_type": "INT", "nullable": true},
        {"name": "name", "data_type": "TEXT", "nullable": true}
      ],
      "file_path": "users.heap",
      "row_count": 0
    }
  }
}
```

このJSONファイルは、データベースディレクトリ内の`catalog.json`として保存されます。

#### カタログの読み込みと保存

```python
def _load(self):
    """カタログをディスクから読み込む"""
    if self.catalog_path.exists():
        with open(self.catalog_path, 'r', encoding='utf-8') as f:
            data = json.load(f)
            for table_name, table_data in data.get('tables', {}).items():
                columns = [
                    ColumnMetadata(**col_data)
                    for col_data in table_data['columns']
                ]
                self.tables[table_name] = TableMetadata(
                    table_name=table_name,
                    columns=columns,
                    file_path=table_data['file_path'],
                    row_count=table_data.get('row_count', 0)
                )
    else:
        # カタログファイルが存在しない場合は空のカタログを作成
        self._save()
```

**実装のポイント**:

1. **起動時の読み込み**: データベースを起動する際、カタログファイルを読み込んでメモリ上に展開します。これにより、高速な名前解決が可能になります。

2. **永続化**: テーブルを作成・削除するたびに、カタログファイルを更新します。これにより、データベースを再起動しても、テーブル情報が保持されます。

### 2. スキーマの定義

スキーマは、テーブルの構造を表現するクラスです。カラム定義のリストと、主キーやインデックスなどの制約情報を保持します。

#### スキーマの構造

```python
class Schema:
    def __init__(self, table_name: str, columns: List[ColumnDefinition]):
        self.table_name = table_name
        self.columns = columns
        self.primary_key: Optional[List[str]] = None  # 将来の拡張
        self.indexes: List[str] = []  # 将来の拡張
```

#### スキーマ検証

スキーマは、値のリストがスキーマに適合するかを検証する機能を持ちます：

```python
def validate_values(self, values: List[any]) -> bool:
    """値のリストがスキーマに適合するか検証"""
    if len(values) != len(self.columns):
        return False
    
    # 型チェック（簡易版）
    for i, (value, col_def) in enumerate(zip(values, self.columns)):
        if value is None and not col_def.nullable:
            return False
        
        # 型チェック
        if value is not None:
            if col_def.data_type == "INT" and not isinstance(value, int):
                try:
                    int(value)  # 変換可能かチェック
                except (ValueError, TypeError):
                    return False
            # ...
    
    return True
```

**実装のポイント**:

1. **カラム数の検証**: 値の数がカラムの数と一致するかを確認します。

2. **型チェック**: 各値の型が、対応するカラムのデータ型と一致するかを確認します。フェーズ1では簡易的な実装ですが、将来的にはより厳密な型チェックを実装します。

3. **NULL値の検証**: カラムが`nullable=False`の場合、NULL値が許可されないことを確認します。

### 3. テーブル作成の流れ

テーブルを作成する際は、以下の手順を踏みます：

1. **SQL文のパース**: `CREATE TABLE`文をパースして、テーブル名とカラム定義を取得
2. **スキーマの作成**: カラム定義から`Schema`オブジェクトを作成
3. **ヒープファイルの作成**: テーブルのデータを保存するヒープファイルを作成
4. **カタログへの登録**: テーブル情報をカタログに登録

```python
def _execute_create(self, stmt: CreateStatement) -> None:
    """CREATE TABLE文を実行"""
    if self.catalog.table_exists(stmt.table_name):
        raise ValueError(f"テーブル '{stmt.table_name}' は既に存在します")
    
    # スキーマを作成
    from .catalog.schema import ColumnDefinition as ColDef
    columns = [
        ColDef(name=col.name, data_type=col.data_type)
        for col in stmt.columns
    ]
    schema = Schema(stmt.table_name, columns)
    
    # ヒープファイルを作成
    file_path = self.db_path / f"{stmt.table_name}.heap"
    heap_file = HeapFile(file_path)
    heap_file.create()
    
    # カタログに登録
    self.catalog.create_table(stmt.table_name, schema, file_path)
    
    # メモリキャッシュに追加
    self.heap_files[stmt.table_name] = heap_file
```

**実装のポイント**:

1. **重複チェック**: 既に存在するテーブル名で作成しようとした場合、エラーを発生させます。

2. **スキーマの作成**: パーサから取得したカラム定義を、`ColumnDefinition`オブジェクトに変換し、`Schema`オブジェクトを作成します。

3. **ヒープファイルの作成**: テーブルのデータを保存するヒープファイルを作成します。ファイル名は`{table_name}.heap`という形式です。

4. **カタログへの登録**: テーブル情報をカタログに登録します。これにより、次回起動時にもテーブル情報が保持されます。

5. **メモリキャッシュ**: ヒープファイルをメモリ上にキャッシュします。これにより、同じテーブルに複数回アクセスする際のパフォーマンスが向上します。

### 4. テーブル情報の取得

テーブル情報を取得する際は、カタログからスキーマを取得します：

```python
def get_schema(self, table_name: str) -> Optional[Schema]:
    """テーブルのスキーマを取得"""
    metadata = self.get_table_metadata(table_name)
    if not metadata:
        return None
    
    columns = [
        ColumnDefinition(
            name=col.name,
            data_type=col.data_type,
            nullable=col.nullable,
            default_value=col.default_value
        )
        for col in metadata.columns
    ]
    
    return Schema(table_name=table_name, columns=columns)
```

**実装のポイント**:

1. **メタデータからの復元**: カタログに保存されている`ColumnMetadata`を、`ColumnDefinition`に変換して`Schema`オブジェクトを復元します。

2. **存在チェック**: テーブルが存在しない場合は`None`を返します。

## 実装のポイント

### 1. JSONファイルによる永続化

フェーズ1では、カタログをJSONファイルで永続化しています。これは、実装が簡単で、人間が読みやすいという利点があります。しかし、将来的には以下の理由から、システムカタログテーブルとして実装する予定です：

- **一貫性**: メタデータもデータとして扱うことで、トランザクション管理が容易になる
- **問い合わせ**: SQLでメタデータを問い合わせることができる（例: `SELECT * FROM pg_tables`）
- **拡張性**: 新しいメタデータを追加する際に、テーブル構造を変更するだけで済む

### 2. メモリキャッシュ

ヒープファイルをメモリ上にキャッシュすることで、同じテーブルに複数回アクセスする際のパフォーマンスが向上します。ただし、現在の実装では、データベースのライフサイクル中にキャッシュが更新されないため、将来的にはより高度なキャッシュ管理が必要になるかもしれません。

### 3. スキーマ検証

INSERT操作の際に、スキーマ検証を行うことで、不正なデータの挿入を防ぎます。フェーズ1では簡易的な実装ですが、将来的には以下のような拡張が可能です：

- **より厳密な型チェック**: 文字列の長さ制限、数値の範囲チェックなど
- **制約の検証**: 主キー制約、外部キー制約、CHECK制約など
- **デフォルト値の適用**: 値が指定されていない場合にデフォルト値を適用

## 実装の全体像：テーブル管理のデータの流れ

最後に、CREATE TABLE操作とINSERT操作のデータの流れを見てみましょう。

### CREATE TABLE操作の流れ

1. **SQL文のパース**: `CREATE TABLE users (id INT, name TEXT)`をパース
2. **スキーマの作成**: `ColumnDefinition`オブジェクトのリストから`Schema`オブジェクトを作成
3. **ヒープファイルの作成**: `users.heap`ファイルを作成
4. **カタログへの登録**: テーブル情報を`catalog.json`に保存
5. **メモリキャッシュへの追加**: ヒープファイルをメモリ上にキャッシュ

### INSERT操作の流れ

1. **SQL文のパース**: `INSERT INTO users VALUES (1, "alice")`をパース
2. **スキーマの取得**: カタログから`users`テーブルのスキーマを取得
3. **値の検証**: スキーマに対して値のリストを検証
4. **タプルの作成**: 値のリストから`HeapTuple`オブジェクトを作成
5. **シリアライズ**: タプルをバイト列に変換
6. **ヒープファイルへの挿入**: ヒープファイルにタプルを挿入

このように、テーブル管理は「メタデータの管理」と「データの管理」を橋渡しする役割を担っています。この基盤があれば、実行エンジンはテーブルの存在や構造を意識せずに、データを操作することができます。

## まとめ

テーブル管理は、データベースの「目録」として、以下の役割を担います：

1. **メタデータの永続化**: テーブル情報をディスクに保存し、再起動後も保持
2. **名前解決**: テーブル名から実際のファイルを見つける
3. **スキーマ検証**: データがスキーマに適合するかを検証
4. **将来の拡張性**: 主キー、外部キー、インデックスなどの機能を追加する基盤

フェーズ1では最小限の機能しか実装していませんが、この基盤があれば、将来的に様々な機能を追加することができます。
